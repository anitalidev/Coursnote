Application Layer

The layer you interact with every day — HTTP, DNS, email. But before the data, there are architectural decisions: who talks to whom?

Open vs Proprietary Protocols

Open Protocols

Publicly documented, usually in IETF RFC documents. Anyone can implement them.

  • HTTP, HTTPS
  • SMTP (email)
  • SSH
  • DNS

Result: many competing implementations, interoperability across vendors

Proprietary Protocols

Private, company-controlled. Only one implementation — yours.

  • Skype
  • iCloud
  • Zoom

Result: platform lock-in, but the company controls the full stack

Network Architectures

Client-Server Architecture

The most common model on the web. Strict roles: the server is always on, always reachable at a known address. The client initiates connections.

  • Server: always on, permanent address/hostname, serves multiple clients at once
  • Client: starts the connection, may be behind NAT or on a changing IP
  • Each connection is strictly one client ↔ one server
  • Examples: web browsing (HTTP), email (SMTP), DNS
Why Client-Server?

Simplicity and reliability. By keeping the server at a known, stable address, clients never have to "find" it. The server can be optimized, secured, and scaled independently. The downside: scaling requires adding more servers (costly), and if the server goes down, so does the service.

Peer-to-Peer (P2P) Architecture

No permanently-on server with a fixed address. Peers (ordinary devices) connect to each other directly and play both client and server roles.

  • Each peer both requests and provides service
  • Self-scaling: each new peer adds demand and capacity
  • Challenges: peers have dynamic IPs, go offline, sit behind firewalls
  • Examples: BitTorrent, early Skype, blockchain networks

P2P Self-Scaling

Imagine 1000 users downloading a file. In client-server, the server bears all the upload load. In P2P, every downloader also uploads to others — the more users, the more capacity. This is why BitTorrent can distribute huge files cheaply.

P2P Address Problem

How do peers find each other if there's no server? Solutions: centralized index (Napster's approach, but a single point of failure), distributed hash tables (DHT, fully decentralized), or tracker servers (BitTorrent's hybrid).

How Chrome Opens a Webpage — Step by Step

When you type https://google.com:

  1. Chrome sees https:// → defaults to port 443
  2. Chrome asks DNS for the public IP of google.com
  3. Chrome now knows the destination IP — tells the OS to send a packet there
  4. The OS checks: is this IP on my local network? (compare first X bits of IP)
  5. No → packet goes to the default gateway (home router)
  6. To reach the router, the OS does an ARP request: "Who has IP 192.168.1.1?"
  7. Router replies with its MAC address
  8. Packet is sent with destination MAC = router's MAC, destination IP = Google's IP
