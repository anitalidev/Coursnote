The Packet Journey

Walk through exactly what happens when your browser opens https://google.com — every address lookup, every hop, every translation.

Click through each step below:

Step 1 of 12
Step 1: Chrome sees "https://"
Chrome parses the URL and sees the https scheme → defaults to port 443. (If you had typed :8010, it would use port 8010 instead.) Chrome now knows what port to contact, but not yet what IP address.
Step 2: DNS Lookup
Chrome asks the DNS resolver (often run by your ISP or a public DNS like 8.8.8.8): "What is the IP address of google.com?" DNS returns, say, 142.250.80.46. Now Chrome knows the destination IP.
Step 3: Is it local?
Your OS compares the first X bits of 142.250.80.46 to your own IP. No match → it's not on your local network. The packet must go to your home router (the default gateway).
Step 4: ARP for the Router
To send a packet to the router, your laptop needs the router's MAC address. It broadcasts an ARP request: "Who has IP 192.168.1.1?" The router replies with its MAC address. (If your laptop already cached it, this step is skipped.)
Step 5: Packet to Home Router
Your laptop sends the packet with:
• Destination IP = 142.250.80.46 (Google — doesn't change end-to-end)
• Destination MAC = your home router's MAC (only valid for this one hop)
Step 6: NAT Translation
Your home router replaces your private IP (192.168.x.x) with its public IP in the packet header. It records this mapping in a NAT table so it can reverse the translation when Google's response arrives. The destination IP stays as Google's.
Step 7: Router Forwarding
The home router checks its forwarding table. Google's IP doesn't match any directly-attached network → it forwards the packet to the ISP router. The destination MAC changes to the ISP router's MAC. The destination IP stays the same.
Step 8: ISP → Backbone
The ISP router uses its routing table (populated by routing protocols like BGP) to forward the packet toward Google's network. Each router along the path swaps in the next hop's MAC address as the destination MAC.
Step 9: Final Delivery
A router whose directly-attached network contains 142.250.80.46 does a final ARP request to get Google's server's MAC address, then delivers the packet directly to Google's server.
Step 10: Google Unpacks
Google's server reads the packet bottom-up through the layers: Physical → Link → Network (IP check) → Transport (port 443 → HTTPS handler) → Application (HTTP request). It processes your request.
Step 11: Response Returns
Google sends a response packet to your public IP. Your home router receives it, checks the NAT table, translates the destination IP back to your private IP (192.168.x.x), and forwards it to your laptop.
Step 12: Your Browser Renders
Your OS receives the packet, the TCP layer assembles the response in order (if using TCP), and hands the HTTP response to Chrome. Chrome renders the page.

Key takeaway: Destination IP stayed the same the whole journey. Destination MAC changed at every single hop.

The Key Invariant

Destination IP — Never Changes

The destination IP is always Google's server IP (142.250.80.46) from start to finish. It's the "final destination" and every router uses it to decide where to forward next.

Destination MAC — Changes Every Hop

The destination MAC is always the next router's MAC. It's overwritten at each hop because MAC addresses only make sense within a single link — they don't route across networks.

Why Two Different Address Layers?

MAC addresses are burned into hardware and work on a single link. They're great for finding "the device right next to me on this cable/WiFi." IP addresses are logical and hierarchical — they're designed for routing across thousands of networks. Using MAC addresses for global routing would fail because there's no geographic hierarchy to MAC addresses — a router couldn't know which direction to forward just from the MAC. IP's hierarchical structure (network portion + host portion) is what makes scalable routing possible.
